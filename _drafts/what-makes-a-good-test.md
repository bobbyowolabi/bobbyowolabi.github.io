In our current age of software engineering <sup>[[a]](#current-age-of-swe-note)</sup>, testing is becoming more and more
of a focus point. If code generation is becoming cheaper and cheaper; verification <sup>[[b]](#ai-software-verification-note)</sup> and quality might be the new coveted 
skills.

Evan King, co-founder of Hello Interview, shared the following on LinkedIn [[1]](#evan-king-citation):

> One engineer at Cloudflare used Claude Code to basically rewrite Next.js from scratch. They reimplemented ~94% of the entire Next.js API surface on top of Vite, called it vinext, and it took about a week for roughly $1,100 in tokens.

How was this accomplished? Instead of reading through the source code line by line, the engineer pointed the AI at Vercel's own test suite and used it as the target. The 2,000+ unit tests and 400+ end-to-end tests effectively became … a specification. The AI's goal was simply to make the new code pass them.

King put it plainly:

> We've always thought of code as the asset and tests as the safety net. Something you write after the fact, even reluctantly. But what this shows is that the tests are actually the more valuable piece now. The code is replaceable.

He concluded: *"The moat isn't in the code anymore. It's in the spec."*

Tests as the specification. Code as the replaceable output. This is a meaningful shift in how we think about software.

There is a growing school of thought that writing code by hand will eventually become as archaic as writing machine code … something specialists peer at, not something the average engineer does day to day. If that is true, then tests become the durable artifact. The thing that outlives any particular implementation.

But here is the complication. The same AI boom that makes tests so powerful also makes it trivially easy to generate *bad* ones. Carlos Arguelles, an engineer who has worked on testing at both Google and Amazon, put it this way [[2]](#carlos-citation):

> do you 𝐭𝐞𝐬𝐭 𝐭𝐡𝐞 𝐭𝐞𝐬𝐭𝐬? Ever heard of 𝐦𝐮𝐭𝐚𝐭𝐢𝐨𝐧 𝐭𝐞𝐬𝐭𝐢𝐧𝐠? In this AI age, it's super easy to have your favorite coding assistant generate hundreds or thousands of unit tests. But some of these tests are subpar to say the least. I caught many unit tests generated by AI that tested absolutely nothing.

So we are in an interesting place. Tests are arguably more valuable than they have ever been, and the tools to generate them carelessly have never been more available.

Given all of this, I thought I'd share what I believe are good qualities of a test. I'll note the irony: this post could become irrelevant in six months given the pace of change. But I like testing, so why not?

A note on scope: I am talking about good old classic unit-like tests — the kind you write in code. There are other verification approaches (simulation, adversarial testing, mutation testing, and more) but that is a conversation for another day.

---

### What Is a Test?

Before getting into qualities, it's worth establishing a mental model. I think of a test as consisting of three things:

1. **Inputs / Environment State** — the conditions you are setting up
2. **Execution** — the call or sequence of operations on the system under test
3. **Assertions** — the verification of the output or resulting state

Everything I look for in a good test flows from how clearly these three things are expressed.

---

### Readable

Tests are written to ensure that a system works. But they are also written for humans … well, for how long we cannot say.

When a test fails, someone has to read it, reason about it, and debug it. That is a real cost. It is worth investing in making that experience as painless as possible.

I love it when the three parts of a test — state, execution, and assertions — are immediately visible. It should be easy to see what conditions are being tested, how the component is being exercised, and what outcomes are being verified. Boilerplate code can obscure all of this. Setup, teardown, plumbing … all necessary, but distracting when it sits right alongside the logic you actually care about. The solution is abstraction — extract the noise into well-named helpers so the body of the test communicates its intent cleanly.

I've been making heavy use of the [Builder pattern][builder-pattern-link] for this lately. For expressing test state, a fluent builder reads almost like a sentence:

```java
User user = new UserBuilder()
    .withRole(Role.ADMIN)
    .withAccountStatus(AccountStatus.ACTIVE)
    .withEmailVerified(true)
    .build();
```

The same approach works well for assertions. Rather than writing raw comparison logic inline, I wrap assertion logic into well-named helpers that can be chained together. The intent becomes visible without requiring the reader to parse the mechanics.

A readable test is one where the next engineer — or your future self — can understand what is being verified without having to reconstruct the context from scratch.

---

### Reliable & Trustworthy

A test that sometimes fails for no good reason is worse than no test at all.

When engineers learn that a test suite produces false positives — failures that don't indicate real problems — they stop trusting it. They start dismissing red builds. They merge anyway. It is the software equivalent of the boy who cried wolf … eventually no one comes running, even when it matters.

This is especially pronounced in end-to-end and integration tests, where timing dependencies, network conditions, and shared state conspire to produce intermittent failures. But it can happen in unit tests too, through poorly isolated state or time-dependent logic.

Reliability also shows up in a subtler way: the experience of reading a failure summary. A trustworthy test suite is one where, when you see a failure, your first instinct is *something is actually wrong* — not *is this another flaky one?* That confidence is earned slowly and lost quickly. It is worth protecting.

---

### Actionable

When a test fails, it should tell you something useful. Not just *that* something broke … but *what* broke, and *where to start looking*.

Consider a test that fails like this:

```
java.io.IOException: An error occurred
    at com.example.service.DataProcessor.process(DataProcessor.java:42)
    at com.example.service.DataProcessorTest.testProcess(DataProcessorTest.java:18)
```

This tells you almost nothing. Was the file missing? Was the connection refused? Was a permission denied? You are starting from square one.

Now compare that to:

```
java.io.IOException: Failed to read config file: /etc/app/config.json (Permission denied)
    at com.example.service.DataProcessor.process(DataProcessor.java:42)
    at com.example.service.DataProcessorTest.testProcess(DataProcessorTest.java:18)
```

Now you have a starting point. The exception message is doing real work.

I go to lengths to make sure exception messages in my code are specific and contextual — not just *what* went wrong, but *where* and with *what input*. This matters especially when the same exception type can be thrown for multiple reasons at different branches. A generic `IOException` thrown in three different places with the same message makes debugging feel like archaeology.

The same philosophy applies to assertion messages. When an assertion fails, the default output from most frameworks is serviceable but often not enough. A meaningful failure message — one that shows what was expected, what was actual, and ideally a hint about what was being checked — can save the next person real time. That next person will often be you.

A test that fails loudly and clearly is a gift. A test that fails cryptically is a tax.

---

King's closing line keeps coming back to me: *"The moat isn't in the code anymore. It's in the spec."*

If that is true — and the vinext story makes a compelling case that it is — then the quality of a test is not a nice-to-have. It is the quality of your entire system. A readable, reliable, and actionable test suite is one that can serve as the blueprint for whatever comes next.

It is worth building it like it matters … because increasingly, it is the thing that does.

How does Vercel come into play in the story?
Because Next.js is open source, Vercel's entire test suite for it is publicly available. The Cloudflare engineer didn't have to write tests from scratch or reverse-engineer the framework by reading source code. They simply pointed the AI at Vercel's own tests — the ones Vercel wrote to verify their own framework — and used those as the specification.
So Vercel is essentially the unwitting blueprint provider. They wrote 2,000+ unit tests and 400+ E2E tests to validate Next.js, published them openly, and that test suite became the target the AI had to satisfy when rebuilding the framework on a completely different foundation (Vite instead of whatever Next.js runs on).
The irony King is pointing at:
Vercel open-sourced their code and their tests, not thinking much of it. But in an AI world, the tests turned out to be the more valuable artifact — a machine-readable spec that let a competitor essentially clone their product in a week for $1,100. That's why King raises the question about whether frameworks will start keeping test suites proprietary going forward.
